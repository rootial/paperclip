# Cabinet research

Date: 2026-04-03

Sources:
- https://runcabinet.com/
- https://github.com/hilash/cabinet

## Snapshot

Cabinet is positioning itself as an "AI-first knowledge base and startup OS": local-first, markdown-on-disk, git-backed, self-hosted, and centered on Claude-driven agents.

From the current public repo and site, the strongest implemented ideas look like:

- a very sharp onboarding promise: create an AI team from a short wizard
- a filesystem-native knowledge base with markdown, PDFs, CSVs, and embedded HTML apps
- durable agent conversations with saved prompts, transcripts, summaries, and artifact paths
- live PTY-backed agent sessions in the browser, not just headless runs
- simple cron-based jobs and heartbeat scheduling
- auto git history, diff, restore, and sync affordances around work products
- a "gallery" style view of recent agent outputs

As of 2026-04-03, the repo is very new and moving quickly. The repo metadata shows it was created on 2026-04-03, and there is already some product drift between the README/site and the current codebase notes in `PROGRESS.md` (for example chat/missions are still marketed in places after being removed in the implementation). That is useful signal in itself.

## What Cabinet gets right

### 1. Strong product framing

Cabinet sells an outcome, not a mechanism:

- "your knowledge base"
- "your AI team"
- "from zero to AI team in 2 minutes"

Paperclip is stronger on governance and real company structure, but Cabinet's framing is easier to understand in one pass. Their marketing compresses the value proposition far better than our current control-plane language.

### 2. Work products are first-class

Cabinet treats files, pages, apps, transcripts, and reports as the main thing the user sees. The system is organized around visible outputs in the workspace, with recent artifacts easy to browse.

This aligns with Paperclip's "output-first" design goal. Cabinet is a good reminder that users want to inspect results before they care about orchestration internals.

### 3. Durable conversation artifacts

The conversation store is a strong pattern: each run has prompt, transcript, trigger, summary, context summary, and artifact paths. That makes agent work replayable and inspectable without forcing the user to read raw terminal output.

Paperclip already has runs, issues, comments, and activity, but Cabinet's conversation packaging is cleaner for "what happened in this run?".

### 4. Embedded app preview is compelling

The "drop an `index.html` in a folder and render it inline" idea is simple and good. It turns agent output into something interactive immediately, especially for generated prototypes, reports, calculators, and dashboards.

For Paperclip, this matters because a lot of agent work already ends in files inside workspaces. Cabinet gives a concrete example of how to surface those files in a board-friendly way.

### 5. Scheduling UX is simple

Cabinet uses plain cron under the hood but wraps it in a more legible workflow: heartbeat schedules, job schedules, presets, human-readable labels, and a daemon that reloads schedules when agent files change.

Paperclip's execution model is more rigorous, but the authoring UX for recurring work could be much simpler.

## Where Cabinet looks weaker than Paperclip

### 1. Governance is thin

Cabinet is optimized for a single operator working in a local workspace. It does not appear to have Paperclip's stronger company model around approvals, assignment semantics, budgets, actor boundaries, and auditable task control.

This means it is easier to start, but much less suited to a real control-plane role.

### 2. Product surface is volatile

The repo's `PROGRESS.md` shows major feature additions and removals on the same day the repo was created. README/site positioning is already ahead of or out of sync with some implementation details. That suggests fast taste and momentum, but also unstable contracts.

Paperclip should copy the clarity and speed, not the churn.

### 3. It is more "AI workspace" than "company operating system"

Cabinet feels closer to a local AI-native workspace or knowledge base with agents layered in. Paperclip's differentiation is the opposite direction: company first, governance first, work delegation first.

We should not blur that distinction by turning the Paperclip core into a note-taking app.

### 4. Git automation is nice but risky

Auto-commit, auto-pull on load, and direct restore are convenient, but they also create noisy history and hidden state transitions. For Paperclip, that behavior should stay explicit or be clearly opt-in at the workspace level.

## Features worth stealing for Paperclip

### High-value

1. Workspace artifact gallery
   A board-visible gallery of recent outputs across issues/agents/projects: docs, screenshots, HTML apps, CSVs, reports, previews.

2. Run replay bundle
   For each run, show a compact bundle: summary, trigger, transcript, artifacts, output links, and maybe "files changed". This should sit above raw logs.

3. Embedded HTML/app preview
   If an issue workspace contains an app-like artifact, render it directly in Paperclip with a safe preview surface.

4. Better first-run company bootstrap
   Cabinet's 5-question setup is a good pattern. Paperclip should have a similarly compressed path from install to "my CEO/company did something real".

5. Human-readable recurring-work authoring
   Keep our existing routine model, but add presets, natural labels, and a tighter editing surface.

### Medium-value

1. Agent output pages / personal workspace directories
   Give each agent a more legible "home" for artifacts and decisions, even if the canonical control surface remains issues/comments.

2. Diff / restore views for workspace files
   Especially useful when agents edit docs, prompts, plans, or generated site assets.

3. Mentionable workspace context in direct agent prompts
   Cabinet's page mentions are a good pattern for passing focused context without loading an entire project.

## What I would not copy

- filesystem-as-primary-data-model for the Paperclip core
- password-only / local-only assumptions
- Claude-specific coupling
- making chat or a KB the center of the product
- silent git side effects by default

## Net

Cabinet is not strongest where Paperclip is strongest. It is strongest where Paperclip is still relatively weak:

- packaging
- onboarding
- work-product visibility
- replayability of agent output
- local workspace delight

The best move is not to become Cabinet. It is to add a few Cabinet-style output and UX surfaces on top of Paperclip's stronger company/governance model.
